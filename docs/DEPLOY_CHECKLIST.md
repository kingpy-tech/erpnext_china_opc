# DEPLOY_CHECKLIST.md - GitHub Pages 首次部署人工验收清单

> 这份清单是给 **首次 GitHub Pages 部署完成后** 用的。
> 不是教你怎么配 workflow，而是帮你确认：**Actions 跑完了不等于真的上线没翻车**。
>
> 适用对象：当前项目的 MkDocs 文档站，部署方式为：
>
> - 工作流文件：`.github/workflows/docs.yml`
> - 部署命令：`mkdocs gh-deploy --force --clean`
> - 发布目标：`gh-pages` 分支

---

## 一、上线前提确认

在开始人工验收前，先确认下面这几件事已经成立：

- [ ] 已将代码推送到 `main` 分支
- [ ] 本次提交命中了 workflow 触发路径（如 `docs/**`、`mkdocs.yml`、`.github/workflows/docs.yml`）
- [ ] 仓库中已存在 `.github/workflows/docs.yml`
- [ ] `mkdocs.yml` 中的 `site_url` 已配置为：

```yaml
site_url: https://kingpy-tech.github.io/erpnext_china_opc/
```

- [ ] 仓库 Settings → Pages 已确认 Pages 来源策略与当前 workflow 方案一致

> 备注：本项目当前 workflow 使用的是 **`mkdocs gh-deploy` 直接推送 `gh-pages` 分支**，所以验收重点是：
> 1. Actions 是否成功
> 2. `gh-pages` 分支是否生成
> 3. 页面是否真能访问、资源是否没掉、导航是否没断

---

## 二、Actions 工作流验收

### 1. 检查 Actions 是否触发
进入 GitHub 仓库：

- `Actions` 标签页
- 找到工作流：**Deploy MkDocs to gh-pages**

验收项：
- [ ] 本次 push 后成功触发 workflow
- [ ] 触发来源正确（`push` 到 `main`，或手动 `workflow_dispatch`）
- [ ] 运行记录中没有出现“skipped”导致未部署的情况

### 2. 检查 workflow 是否执行成功
点开最新一次运行记录，确认以下步骤全部为绿色：

- [ ] `Checkout repository`
- [ ] `Setup Python`
- [ ] `Install dependencies`
- [ ] `Build documentation`
- [ ] `Deploy to gh-pages branch`

### 3. 重点看日志里有没有这些翻车信号
如果构建虽然“结束了”，但日志里有下面这些东西，也要当问题看：

- [ ] 没有出现 `mkdocs build --strict` 构建失败
- [ ] 没有出现 `Unrecognised theme name 'material'`
- [ ] 没有出现 `Config value 'site_url'` 相关报错
- [ ] 没有出现权限相关错误（如 push `gh-pages` 被拒绝）
- [ ] 没有出现路径错误、文档缺失、YAML 格式错误

### 4. 失败时优先排查方向
如果 Actions 红了，优先看这几个方向：

1. `mkdocs.yml` 语法或 nav 配置错误
2. `mkdocs-material` 未正确安装
3. 文档内部链接/引用文件路径错误（`--strict` 容易把问题直接打出来）
4. GitHub Actions 权限不够，导致无法推送 `gh-pages`

---

## 三、gh-pages 分支验收

### 1. 检查分支是否生成
进入仓库分支列表，确认：

- [ ] 已存在 `gh-pages` 分支
- [ ] `gh-pages` 分支更新时间与最新部署时间一致
- [ ] `gh-pages` 分支不是空分支

### 2. 检查静态产物是否存在
打开 `gh-pages` 分支，确认至少存在以下典型文件/目录：

- [ ] `index.html`
- [ ] `404.html`（如果构建产物中有）
- [ ] `assets/` 或等价静态资源目录
- [ ] `search/` 或搜索索引相关文件（若启用了搜索）
- [ ] CSS / JS / 图片等资源文件已生成

### 3. 检查 gh-pages 内容是否像“真实网站”而不是半拉子工程
- [ ] `index.html` 能打开，不是空文件
- [ ] 页面引用的静态资源路径不是明显错误路径
- [ ] 分支里不是只有源码文档 `.md`，而是已构建好的静态站点

---

## 四、页面访问验收

### 1. 主页是否可访问
访问：

- `https://kingpy-tech.github.io/erpnext_china_opc/`

验收项：
- [ ] 页面能正常打开
- [ ] 不是 GitHub 默认 404 页面
- [ ] 不是空白页
- [ ] 页面标题、站点名称正常显示
- [ ] 首屏渲染正常，没有明显布局错乱

### 2. 404 页面验收
人工访问一个不存在的路径，例如：

- `https://kingpy-tech.github.io/erpnext_china_opc/not-found-test`

验收项：
- [ ] 返回结果符合预期
- [ ] 若项目配置了自定义 404 页面，则应显示项目自己的 404
- [ ] 若没有自定义 404，也至少要确认不是因为基础路径错误导致整个站都 404

> 这里的重点不是“404 页面一定要多华丽”，而是要确认：
> **404 是个别路径不存在，而不是整站资源路径炸了。**

### 3. 页面刷新与深链接验收
任选几个二级页面直接访问或刷新，例如：

- 文档首页
- ERPNext 快速开始页
- 配置/部署相关页面

验收项：
- [ ] 通过导航点击进入页面正常
- [ ] 复制二级页面 URL 直接打开也正常
- [ ] 浏览器刷新后二级页面不出现异常 404

---

## 五、静态资源验收

这是 GitHub Pages 最容易“表面上线、实际裸奔”的地方。

### 1. 样式是否加载正常
- [ ] 页面样式正常，不是纯 HTML 裸页
- [ ] 字体、颜色、间距、按钮样式正常
- [ ] 代码块样式正常显示
- [ ] 顶部导航、侧边栏样式正常

### 2. JS 是否加载正常
- [ ] 菜单展开/折叠可用
- [ ] 搜索功能可正常打开（若启用）
- [ ] 移动端菜单可正常展开
- [ ] 主题切换/交互按钮可正常工作（若启用）

### 3. 图片 / 图标 / favicon 是否正常
- [ ] Logo 正常显示
- [ ] favicon 正常显示
- [ ] 文档内插图无 broken image
- [ ] 社交图标、仓库图标等正常显示

### 4. 浏览器控制台检查
打开浏览器开发者工具，检查：

- [ ] 没有大面积 404 静态资源报错
- [ ] 没有明显的 CSS / JS 加载失败
- [ ] 没有因为 base path 错误导致资源从根路径 `/` 去加载

> 如果控制台里全是 `/assets/... 404`，通常就是 `site_url`、相对路径或 Pages 基础路径没配对。

## 六、导航与内容验收

### 1. 顶部/侧边导航检查
- [ ] 顶部导航可以正常点击
- [ ] 侧边导航层级展开正常
- [ ] 当前页面高亮正常
- [ ] 返回首页路径正确

### 2. 关键文档抽查
至少抽查以下页面：

- [ ] 首页 `docs/index.md` 对应页面
- [ ] `docs/erpnext/00_Quick_Start.md`
- [ ] `docs/WHY_CONTRIBUTE.md`
- [ ] `README` 中提到的文档站入口页

### 3. 页面内链接检查
- [ ] 页面内锚点跳转正常
- [ ] 页与页之间的内部链接正常
- [ ] 指向 GitHub 仓库的外链正常
- [ ] 没有明显“点进去就是 404”的死链

## 七、首次部署后的结论记录模板

建议在首次部署完成后，把结果记进 `HANDOFF/HANDOFF.md` 或发布记录中。

可直接参考下面这个模板：

```md
### GitHub Pages 首次部署验收结果
- Actions: ✅ 成功 / ❌ 失败
- gh-pages 分支: ✅ 已生成 / ❌ 未生成
- 主页访问: ✅ 正常 / ❌ 异常
- 404 页面: ✅ 符合预期 / ❌ 需修复
- 静态资源: ✅ 正常 / ❌ 存在 CSS/JS/图片丢失
- 导航链接: ✅ 正常 / ❌ 存在死链
- 结论: 可上线 / 需修复后重试
```

## 八、常见翻车点速记

### 1. Actions 绿了，但页面还是 404
常见原因：
- `gh-pages` 分支没生成成功
- GitHub Pages 设置没有指向正确来源
- 页面还在缓存，或首次发布尚未生效

### 2. 页面能开，但样式全没了
常见原因：
- `site_url` 基础路径不对
- 资源引用到了根路径 `/`，而不是仓库子路径 `/erpnext_china_opc/`

### 3. 首页能开，二级页面一刷新就炸
常见原因：
- 路由/静态路径处理不一致
- 404 页面或基础路径配置不完整

### 4. 导航能点，但部分页面丢失
常见原因：
- nav 配置没更新
- 文档文件改名后链接没改
- 构建严格模式没覆盖到某些手写链接问题

---

## 九、结论

这份清单的核心目标只有一句话：

**别把“工作流跑完了”误判成“文档站真的稳了”。**

首次上线至少要人工确认四件事：

1. **Actions 成功**
2. **`gh-pages` 分支真的生成**
3. **页面访问正常，不是整站 404**
4. **静态资源和导航都没掉链子**

把这四关过了，GitHub Pages 才算真的站起来，而不是只是看起来像站起来了。
