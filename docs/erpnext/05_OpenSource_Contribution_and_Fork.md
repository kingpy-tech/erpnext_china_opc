# 05 ERPNext 极客篇：如何优雅地魔改、共存与回馈开源社区

## 痛点切入：被无情覆盖的“魔改”代码
很多刚接触开源 ERP 的玩家，在使用的过程中如果遇到了一个错误（比如某个按钮翻译错了，或者某段 Python 逻辑不太符合中国国情），他们的第一反应是：**直接通过 SSH 连进服务器，找到那个源文件，硬生生把代码给改了**。

这虽然能解燃眉之急，但却埋下了一个巨大的地雷：**一旦官方发布了新版本（比如你执行了 `bench update`），你辛辛苦苦修改的代码就会瞬间灰飞烟灭，被官方的原版文件无情覆盖！** 或者更惨的情况，你的硬改直接导致升级脚本崩溃，系统彻底瘫痪。

如何才能让你的修改长久存活？如何与官方的新功能共存？甚至，如何让你随手的修复造福全人类？
这就是开源世界的终极生存法则：**提 PR、建 Fork 库、以及理解 Frappe 框架的解耦哲学**。

---

## 境界一：回馈社区 —— 向官方提交 PR (Pull Request)

如果你发现的仅仅是官方的一些小 Bug（比如某个词翻译不准、某段简单的 UI 代码写错了），最好的解法是不要留在自己手里，而是**直接去敲 GitHub 的大门，把修改教给老外**。这样不仅一劳永逸，下个版本大家都能用上，你的名字还会被永远刻在 ERPNext 的贡献者名单上。

就算你不懂敲复杂的 Git 命令，只需在网页上点点鼠标即可完成：

1. **注册并登录 GitHub**，搜索并进入目标官方仓库（如 `https://github.com/frappe/erpnext`）。
2. **复印仓库（Fork）**：点击右上角的 **`Fork`** 按钮。
   > **🚨 致命防坑警告**：弹出的界面里，请**务必取消勾选** `Copy the develop branch only`！如果你勾了，你的库里就只剩一个测试分支。而我们的云服务器是靠拉取 `version-16` 这种稳定分支活着的。一旦漏掉其他分支，后续你的云服务器就会拉取代码失败直接宕机！
3. **切换到正确的“潜规则”分支**：这是开源界最高阶的规矩。Frappe 官方规定，任何提交的 PR 都必须基于 `develop` 分支。所以你要确保网页左上角的分支下拉框选中的是 **`develop`**。
4. **网页上直接改代码**：进入你自己的那个克隆仓库的 `develop` 分支，找到有错别字的那个文件（比如 `erpnext/translations/zh.csv`），点击右上角的 **✏️ (铅笔图标)** 进入编辑模式，直接在网页里把错别字改对！
5. **保存修改（Commit）**：把网页拉到底部，在 **Commit changes** 输入框里用英文简单描述你干了啥（如 `fix: correct several Chinese translations`），然后点击绿色的 Commit 按钮保存。
6. **发起合并请求（Pull Request）**：回到你仓库的首页，点击 **`Contribute`** -> **`Open pull request`**。这等同于你拿着改好的文件去敲官方的门。点击绿色的 **Create pull request**，接下来就等官方的老外开发者核实后点合并（Merge）啦！

## 境界二：割据一方 —— 维护你自己的私人魔改版 (Sync Fork)

很多时候，你写了一段非常具有“中国特色”的代码，官方出于全球通用性的考量拒绝了你的 PR；或者你压根就不想公开你的商业机密代码。
**如果你依然想享受官方日新月异的新功能，同时又要强行保留你自己的“魔改”代码，该怎么办？**

答案是：**自己做主，让你的云服务器订阅你私人的克隆仓库（Fork 库），而不是官方仓库！**

### 1. 明白“两头改”的潜规则
在**境界一**中我们说了，你要提 PR 给官方，必须改 `develop` 分支。
但是，如果你是想把修改立刻用到自己的公司生产环境上，因为我们的云主机配置拉取的是 **`version-16` 稳定分支**，所以你的服务器根本看不到 `develop` 分支的代码！
**现实中的开发者都是这么干的（两头改）**：为了造福社会，在 `develop` 提个 PR；为了自己公司，切到 `version-16` 分支把刚才的代码再改一遍。

### 2. 把生产环境切换到你的 Fork 库
因为你刚才已经在 GitHub 点过 Fork 并且在你的 `version-16` 分支里写好了你的魔改代码，你拥有了（例如 `kingpy-tech/erpnext`）这个私人仓库。
现在，你进到你的云服务器终端，用自己的仓库地址去替代官方地址：
```bash
# 以后只从你的私有库拉取系统
bench get-app https://github.com/kingpy-tech/erpnext.git
```

### 3. 利用“吸星大法”跟进官方升级 (Sync Fork)
过了一个月，官方发布了很牛逼的新版本。但你个人的 `kingpy-tech/erpnext` 还是老古董。

> **💡 原理防坑：官方更新了，我的 Fork 库会自动更新吗？**
> **绝对不会。** GitHub 故意这样设计是为了保护你。如果官方每天的更新都自动覆盖你的服务器，万一哪天发了个有 Bug 的版本或者跟你的魔改代码冲突了，你的生产环境就会突然崩溃。**系统升级的按钮必须捏在你自己手里。**

当你觉得有空且准备好升级时，你只需要在 GitHub 网页上，进入你自己的仓库首页（注意左上角选对你正在使用的如 `version-16` 分支）：
- 页面上方会提示你：“This branch is out-of-date...”（你的分支落后于官方了）。
- 旁边有一个按钮叫 **`Sync fork`**（同步分支）。点它！
- GitHub 的后台服务器会瞬间发动魔法，把官方新写的一万行代码，和你上个月在这个分支里改的那几行代码完美地融合在一起！你的仓库瞬间就升级到了官方的最新版，并且你的魔改完好无损。

### 4. 拉回服务器
最后，去你的云服务器终端敲一句：
```bash
bench update --patch
```
系统就会去你的私人仓库里，把那个**“官方最新功能 + 你的魔改版”**的融合包拉下来更新。完美！

## 境界三：直面终极 Boss —— 解决代码冲突 (Merge Conflict)

在使用 **境界二（Sync Fork）** 的时候，你绝对有概率遇到这个让人又爱又恨的红色报错：**"There are merge conflicts"（存在代码冲突）**。

### 为什么会冲突？
假设在你魔改的文件里：
- **你（上个月）**：把第 10 行的按钮颜色改成了`红色`。
- **官方（今天）**：觉得这个按钮没用，直接把第 10 行删了，换成了一个下拉菜单。
当你今天点 `Sync fork` 时，Git 引擎傻眼了：这第 10 行到底该听你的留着红色按钮，还是听官方的删掉变菜单？这就是冲突。

### 怎么解决冲突？（网页断案法）
如果只是一两行代码的冲突，你可以直接在 GitHub 网页上当“法官”：
1. 点击报错提示中的 **Resolve conflicts**（解决冲突）按钮。
2. 页面会展示冲突的代码块。冲突的地方会被极其刺眼的符号包围：
   ```text
   <<<<<<< kingpy-tech:user_profile.py
   这里是你魔改的代码（红色按钮）
   =======
   这里是官方新写的代码（下拉菜单）
   >>>>>>> frappe:version-16
   ```
3. 做一道删减题：在网页编辑器里，**把你认为不对的代码，连同 `<<<<<<<` 和 `=======` 这些分界符统统删掉！** 只留下那行你认为最终应该存在的正确代码。
4. 删完后，点击右上角的 **Mark as resolved**（标记为已解决），再点击 **Commit merge**。你的仓库就又恢复健康了。

## 终极道法：Frappe First（解耦定制哲学）

通过上面的折腾，你发现没？**只要你硬改了底层源码，以后每次升级都有大概率引发代码冲突地狱。**

为了跳出这个无休止的“修冲突”循环，Frappe 框架提供了一条终极的“防冲突”阳关大道，这就是我们在其他教程里反复强调的 **"Frappe First" 原则**：

- **想要修改界面的错别字？** 不去改源文件，用网页上的 `Translation` 数据导入。
- **想要给表单增加字段？** 不去改源表，用网页上的 `Customize Form` 加自定义字段。
- **想要一段全新的业务逻辑？** 不去改官方的 Python 逻辑，而是用 `bench new-app` **自己建一个独立的 App**，然后在自己的 App 里利用 Frappe 强大的 **挂载 (Hooks)** 机制去重写或者拦截官方的逻辑！

当你真正掌握了写独立 App 的解耦思维，你就不再是那个天天担心升级被覆盖、被代码冲突折磨的新手了，而是一个能在一套核心系统上衍生出千变万化玩法的架构大师！
