# 02 ERPNext 高级篇：账套的快照回退与多租户实验室玩法

## 痛点切入：害怕改废系统？
很多初学者在搭建完 ERPNext 后，最常见的心理负担就是：“我想装个新插件”、“我想在财务表单里加减几个字段测试一下逻辑”，但我**害怕把已经装好并运行着真实业务数据的系统给搞崩溃了**。

传统软件由于业务逻辑与数据库强耦合，测试环境和生产环境往往需要两台完全独立的服务器。但在 Frappe 框架下，这完全是多余的担心。借助其强大的**多租户架构（Multi-tenant）**，我们能在同一台云服务器上，近乎**零成本、零额外内存消耗**地开启无数个独立的“实验室账套”。

结合框架自带的**一键备份与恢复（快照）**功能，你可以实现“无限试错、随时回档”的极客玩法。

---

## 玩法一：利用 Hosts 魔法搭建极客“沙盒实验室”

假设你已经有了一个跑着真实业务的内部正式账套（比如绑定在企业 VPN 的 IP `100.x.x.x` 上）。千万不要在上面做未知实验！
我们可以在同个容器里，利用命令创建一个专门用于乱折腾的“沙盒”，并且**连买域名都不需要**。

### 1. 创建一个无歧义的“假域名”账套
既然是你一个人的沙盒，你可以在系统里随便捏造一个名字。
**强烈建议：使用 `.local` 结尾的假域名（例如 `sandbox.local`）**。
这是一种“防呆机制”，当你以后在浏览器看到 `.local` 这个奇怪的后缀时，你的大脑会立刻反应过来：“哦，我现在是在本地特供的沙盒里，可以随便乱搞！”，从而彻底避免把测试站和正式站（如 `qingpy.cn`）搞混。

进入你的 Docker 容器：
```bash
docker exec -it erpnext-backend-1 bash
```

执行建站命令：
```bash
# 1. 建立空数据库
bench new-site sandbox.local --admin-password admin --db-root-password admin

# 2. 安装核心 ERPNext 应用
bench --site sandbox.local install-app erpnext

# 3. (可选) 秒装其他第三方插件。由于底层代码在正式环境部署时已经拉取编译过了（如 hrms），
# 这里无需再 get-app，直接一句话就能挂载给沙盒！
bench --site sandbox.local install-app hrms
```

### 2. 电脑本地修改 hosts 进行魔法穿透访问
**踩坑警告**：很多新手会想着直接把云服务器的物理内网 IP（比如腾讯云的 `10.0.0.2`）当做 Site 名字。结果发现在家里的电脑浏览器死活打不开。因为 `10.0.x.x` 属于云机房内部，你家里的路由器根本找不到它！

**正确的极客解法：**
使用你的**企业 VPN 等虚拟组网工具能够直连云主机的那个 IP**（假设是 `100.119.x.x`），并让你的个人电脑强行认为 `sandbox.local` 就是这个 IP。

修改你个人电脑本机的 `hosts` 文件：
- **Mac/Linux** 打开终端执行：`sudo nano /etc/hosts`
- **Windows** 用管理员权限打开记事本，编辑：`C:\Windows\System32\drivers\etc\hosts`

在文件最末尾，加上这一行映射（前面是你的 VPN 直连 IP，后面是你刚才编的沙盒名字）：
```text
100.119.x.x    sandbox.local
```

### 3. 开始折腾
此时，由于本地 hosts 的拦截，当你在浏览器地址栏敲下 `http://sandbox.local:8080` 时，请求会被你的 VPN 通道瞬间安全地送到云服务器。而云上的 Frappe 框架一看请求头是 `sandbox.local`，立刻完美地将你分配进了这个独立隔离的测试系统！
从此，你可以在这个沙盒里怎么乱删表单、测试错误代码，都不会有一丝一毫影响到你的正式账套了。

## 玩法二：随时随地打“快照”（备份）

在你的沙盒实验室里，或者甚至在你真实的业务账套准备进行大规模改动（比如导入几千条历史财务记录）前，**一定要养成打快照的习惯**。

打快照（备份）在 ERPNext 中极其简单，一条命令就能将当前账套的数据库和所有配置文件压缩备份：

```bash
# 假设你在操作内部生产库 10.0.0.2，请将其替换为你要备份的 site 名字
bench --site 10.0.0.2 backup
```

执行完毕后，系统会提示备份成功，并将数据打包存放在 `/home/frappe/frappe-bench/sites/10.0.0.2/private/backups/` 目录下。你会看到类似下面这样名字的文件：
- `20240101_120000-10_0_0_2-database.sql.gz` (数据库快照)
- `20240101_120000-10_0_0_2-site_config_backup.json` (配置快照)

## 玩法三：玩崩了？一键回档（恢复快照）

这就是 ERPNext 极度好用的“读档”机制。假设你在刚才测试导入单据时搞出了一堆无法删除的垃圾账目，想要回到十分钟前你敲 `backup` 时的完美状态。

只需执行 `bench restore` 命令，并跟上你刚才备份出的 `.sql.gz` 文件的绝对路径：

```bash
# 注意：这会抹除当前该账套的所有数据，并用备份文件覆盖！
bench --site 10.0.0.2 restore /home/frappe/frappe-bench/sites/10.0.0.2/private/backups/20240101_120000-10_0_0_2-database.sql.gz
```

短短几秒，系统数据就会像时光倒流一样，完美恢复到你备份的那一刻。

## 玩法四：彻底推翻，销毁重建（Drop Site）

如果你在沙盒实验室里装了一堆奇怪的 App，改得面目全非，连回档都不想回了，觉得系统太乱。
你可以选择最暴力的玩法：**直接抹除整个租户，重新做人**。

执行这条毁灭性命令（**高危操作，请再三确认你敲的 site 名字是你不要的沙盒，而不是正式库！**）：
```bash
bench drop-site sandbox.local --force
```

这条命令会直接从 MariaDB 中删掉这个租户所有的数据库表，并清空所有上传的附件文件夹。随后，你可以重新运行 `bench new-site sandbox.local` 重建系统。

**踩坑提醒（出现 Internal Server Error 怎么办？）：**
由于 ERPNext 底层的 Gunicorn 后端服务和 Nginx 前端会把站点的状态缓存在内存中。有时你在直接 `drop` 掉一个站并立刻同名重建后，浏览器访问可能会抛出 `Internal Server Error (500)`。
这并不是你建站失败，**只需去 1Panel 的容器编排页面，将整个 `erpnext` 集群重启一次（或者在终端执行 `docker restart erpnext-frontend-1 erpnext-backend-1`）**，让服务重新读取配置即可恢复正常！

## 玩法五：高阶克隆——将生产环境数据导入沙盒模拟

这是企业级部署中最实用的一招！比如到了年底，你要对系统中几千条凭证进行“年结转（Year End Close）”，或者第一次算全公司的工资。你敢在真实业务系统里直接点“开始计算”吗？万一参数配错了怎么办？

此时，你可以**把正在运行的真实业务数据，一键克隆到刚才建好的沙盒里**进行演练：

**第一步：给正式账套打个快照**
```bash
# 假设 10.0.0.2 是你内部真实的生产账套
bench --site 10.0.0.2 backup
```

**第二步：把正式账套的快照，跨站恢复给测试账套！**
这句命令的魔法在于：`--site` 指定了你要把数据倒进哪个坑里（填测试站），而后面的文件路径却指向了正式站刚才备份出来的数据。
```bash
# 注意路径中的 10.0.0.2 和最后的文件名，请根据你实际备份出来的内容替换
bench --site sandbox.local restore /home/frappe/frappe-bench/sites/10.0.0.2/private/backups/20240101_120000-10_0_0_2-database.sql.gz
```

**结果：** `sandbox.local` 瞬间变成了你正式系统的 **1:1 完美克隆体**！所有最新的发票、真实客户、你熬夜配好的科目流全部都在里面，且与正式站**物理断开**。你现在可以毫无顾忌地在这个沙盒里测试年结账了，哪怕点错了导致财务数据清零，对真正的 `10.0.0.2` 系统也没有任何影响。

## 总结：用“开发者思维”使用 ERPNext
通过：
1. **多租户建测试库**（物理隔离）
2. **`bench backup`**（存读档）
3. **`bench drop-site`**（销毁重开）

你就拥有了一套属于自己的、容错率极高的企业数字化实验室。无论是测试会计科目初始化、还是演练极其复杂的生产制造 BOM 树，都不再有任何心理负担。大胆去尝试吧！
